\section*{\thesection. Deployment and Maintenance Instructions} \label{app:deployment-and-maintenance-instructions}
\addcontentsline{toc}{subsection}{\protect\numberline{\thesection.}Deployment and Maintenance Instructions}
This appendix provides brief deployment and post-deployment maintenance instructions for the application in the three deployment targets discussed in Chapter~\ref{sec:deployment-environments-and-infrastructure-setup}.
Begin by cloning the repository and ensuring that the required deployment tool is installed, namely Helm\footnote{\url{https://helm.sh/docs/intro/install}} for Kubernetes and Terraform\footnote{\url{https://developer.hashicorp.com/terraform/install}} for Azure and AWS\@.

\begin{minted}{bash}
# Clone the repository
git clone https://github.com/eistre/terminal.git

# Change to the project directory
cd terminal
\end{minted}

Environment-specific credentials, domains, secrets, and provider-side prerequisites must be prepared separately.
The following parts provide instructions for each deployment target.

\subsection*{1. UT HPC / Kubernetes}
To deploy the application in Kubernetes, change to the \path{deploy/kubernetes/} directory.
Fill in either \path{values-hpc.example.yaml} or \path{values-local.example.yaml}, based on whether the deployment is targeting the UT HPC cluster or another Kubernetes environment with the required values.
This deployment target assumes that the Kubernetes namespace and UT HPC-related integrations have already been prepared as described in Chapter~\ref{subsec:university-of-tartu-hpc}.

\begin{minted}{bash}
# Change to the Kubernetes deployment directory
cd deploy/kubernetes

# Apply the Helm chart using the appropriate values file
helm upgrade --install --namespace <NAMESPACE_NAME> -f <VALUES_FILE> terminal .
\end{minted}

This command sequence installs the chart or updates an existing release using the environment-specific values file to the specified Kubernetes namespace.
After deployment, the environment administrator should periodically verify that the application is reachable and that terminal sessions can be started.
If this check fails, the Kubernetes resources created by the Helm release should be inspected.
In the UT HPC deployment, cluster-level infrastructure is operated by UT HPC, so routine work remains mainly namespace-scoped.
The chart uses \texttt{latest} image tags by default and sets \texttt{imagePullPolicy} to \texttt{Always} for such images, meaning that newly started pods pull the current image.
If updated images are published under the same tag and an immediate refresh is needed, the deployment should be restarted.
When needed, the repository maintainer reviews dependency update pull requests prepared by Renovate, merges approved changes, and relies on the repository's automated workflow to rebuild and publish updated container images.

\subsection*{2. Microsoft Azure}
To deploy the application in Microsoft Azure, change to the \path{deploy/azure/} directory.
Copy \path{terraform.prod.tfvars.example} to \path{terraform.tfvars}, then fill in the required values.

\begin{minted}{bash}
# Change to the Azure deployment directory
cd deploy/azure

# Copy the example Terraform variables file to a new file
cp terraform.prod.tfvars.example terraform.tfvars

# Initialize the Terraform working directory
terraform init

# Apply the Terraform configuration using the variables file
terraform apply -var-file=terraform.tfvars
\end{minted}

This command sequence initializes the Terraform working directory and applies the configuration for Azure.
After deployment, Azure operates the underlying managed services, while the environment administrator should periodically check service health, monthly cost, and scheduled cleanup execution.
Application images are controlled through the Terraform variable \texttt{image\_tag}, which defaults to \texttt{latest}.
When the image tag is changed, the deployment can be updated through Terraform.
If updated images are published under the same tag, an immediate refresh requires restarting or redeploying the Container Apps workload.
When needed, the repository maintainer reviews dependency update pull requests prepared by Renovate, merges approved changes, and relies on the repository's automated workflow to rebuild and publish updated container images.

\subsection*{3. Amazon Web Services}
To deploy the application in Amazon Web Services, change to the \path{deploy/aws/} directory.
Copy \path{terraform.prod.tfvars.example} to \path{terraform.tfvars}, then fill in the required values.

\begin{minted}{bash}
# Change to the AWS deployment directory
cd deploy/aws

# Copy the example Terraform variables file to a new file
cp terraform.prod.tfvars.example terraform.tfvars

# Initialize the Terraform working directory
terraform init

# Apply the Terraform configuration using the variables file
terraform apply -var-file=terraform.tfvars
\end{minted}

This command sequence initializes the Terraform working directory and applies the production configuration for AWS\@.
After deployment, AWS operates the managed services used by the environment, while the environment administrator should periodically check service health, monthly cost, and scheduled cleanup execution.
Application images are controlled through the Terraform variable \texttt{image\_tag}, which defaults to \texttt{latest}.
When the image tag is changed, the deployment can be updated through Terraform.
If updated images are published under the same tag, restarting or redeploying the Lightsail container service and ECS workloads is required.
When needed, the repository maintainer reviews dependency update pull requests prepared by Renovate, merges approved changes, and relies on the repository's automated workflow to rebuild and publish updated container images.
