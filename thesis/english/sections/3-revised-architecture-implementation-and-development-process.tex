\section{Revised Architecture and Implementation} \label{sec:revised-architecture-and-implementation}
This chapter outlines the modernized architecture and implementation of the Linux command-line learning environment.
Building upon the baseline system described in chapter~\ref{sec:background-and-previous-system}, it focuses on five main implementation changes that addressed the baseline limitations.
These changes are the modularization of the codebase, the introduction of a provider-agnostic provisioning layer, the separation of cleanup jobs from the main application, the redesign of authentication and single sign-on support, and the redesign of multilingual content management.
Together, these changes improved maintainability, portability, and operational efficiency while preserving the core learning workflow of the platform.

\subsection{Development Process and AI Assisted Refactoring} \label{subsec:development-process-and-ai-assisted-refactoring}
The implementation work in this thesis required coordinated refactoring across multiple layers of the baseline system while keeping the codebase operational throughout the change.
To speed up this process, the author used coding assistants based on large language models (LLMs) as interactive tools for refactor planning, scaffolding, and iterative debugging.
However, final responsibility for design decisions, integration, and verification remained with the author.

\subsubsection{AI Assisted Development} \label{subsubsec:ai-assisted-development}
Recent software engineering research describes LLM-assisted development as an interactive workflow in which developers use natural language to speed up well-understood subtasks and explore implementation options~\cite{ross_programmers_2023,barke_grounded_2023}.
Studies of professional use also report that these tools are valued for reducing typing effort, syntax recall, and online searching, but that their output depends strongly on context and requires careful human review and validation before integration~\cite{davila_industry_2024,barke_grounded_2023}.

In this thesis, AI-assisted coding was used as a support method for planning, scaffolding, boilerplate generation, and debugging.
Architectural decisions, integration, and final acceptance of changes remained the responsibility of the author.

\subsubsection{Refactoring Workflow} \label{subsubsec:refactoring-workflow}
AI assistance was primarily used through OpenCode\footnote{\url{https://opencode.ai/}}, a command-line AI coding tool.
Access to the underlying models was obtained through GitHub Copilot using the GitHub Education student benefit~\cite{noauthor_supported_nodate,noauthor_students_2026}.
The primary models used were Claude Sonnet 4.5~\cite{anthropic_claude_sonnet_4_5_2025} and GPT-5.2-Codex~\cite{openai_gpt_5_2_codex_2025}, selected based on availability and suitability for coding and debugging tasks.

Most of the significant changes, such as the container provisioning abstraction and the localization redesign, followed a consistent workflow:

\begin{enumerate}
  \item Scope and constraint definition: define a concrete implementation goal together with technical constraints and acceptance conditions;
  \item Prompt-driven planning and design exploration: ask the assistant to analyze the existing code and propose a development plan;
  \item Plan iteration: refine the proposed approach through discussion until the intended change scope is clear;
  \item Scaffold and draft code changes: generate boilerplate and first-pass implementations to reduce repetitive manual work;
  \item Review and integration: inspect, simplify, and adapt the generated output before integrating it into the codebase;
  \item Verification and iteration: validate the result through type checking, linting, and manual testing, then use concrete failures as input for follow-up fixes.
\end{enumerate}

One example prompt used for refactoring is shown below:

\begin{minted}{text}
Analyze the existing Kubernetes-specific runtime administration
and provisioning logic in this codebase and propose a refactoring plan
for restructuring it into a modular provisioner layer.

Focus on the following:
- extract an abstract container provisioning interface from the current
  Kubernetes specific pod and environment administration logic;
- define a Kubernetes implementation behind that interface;
- follow the new modular structure used in the repository,
  placing the provisioner package inside the `packages/` directory;
- introduce shared concurrency control, retry handling, and safeguards
  for provisioning operations;
- identify sensible implementation steps for carrying out the refactor.
\end{minted}

Given the risk of integrating incorrect or overly complex generated changes, explicit validation steps were applied.
Large refactors were divided into smaller, reversible steps.
Changes were accepted only after passing type checks, linting, and manual verification.
Additionally, security- and cost-sensitive areas received stricter review.
These measures align with prior work on AI programming assistants, which emphasizes treating generated code as draft material that still requires contextualization, review, and validation before it is integrated into the codebase~\cite{barke_grounded_2023,davila_industry_2024}.

\subsection{Architecture Overview} \label{subsec:architecture-overview}
The revised system architecture separates the interactive learning workflow from infrastructure-specific concerns and periodic maintenance.
This design preserves the user-facing workflow while allowing the execution backend and operational mechanisms to differ across deployments.

\begin{figure}[ht]
  \centering
  \includegraphics[width=\textwidth]{figures/figure5-revised-architecture}
  \caption{Revised architecture overview of the modernized learning environment.}
  \label{fig:revised-architecture-overview}
\end{figure}

Figure~\ref{fig:revised-architecture-overview} groups the system into four conceptual components.
The first is the browser, responsible for rendering the web interface and hosting the interactive terminal.
The second is the main application environment.
It contains the primary Nuxt server and the scheduled maintenance jobs.
In this context, the Nuxt server provides both the Vue.js user interface and the API surface used by the frontend.
The third component is the database, which retains the persistent application state required for continuity and administration, including user data, progress, and educational content.
The fourth component is the container execution environment, which provides isolated Linux sessions for users during terminal-based exercises.

A significant architectural change compared to the baseline system is the separation of maintenance tasks from the primary application into dedicated runnable functions deployed as scheduled jobs.
As illustrated in Figure~\ref{fig:revised-architecture-overview}, these are represented by the provisioner cleanup job and the database cleanup job.
The provisioner cleanup job enforces lifecycle policies for user container environments, while the database cleanup job removes expired verification tokens and inactive user records in accordance with retention policies.
This separation means that the main application does not need to run indefinitely only to perform periodic maintenance.
It also allows deployments where the main application can be scaled down during idle periods while maintenance continues to preserve system consistency.

During an active terminal session, the browser establishes a persistent connection to the backend API to support interactive input and output.
The backend then bridges this traffic to the user's Linux session in the container execution environment.
The details of how the user session is provisioned and how lifecycle rules are implemented are covered in later subchapters.

This overview remains provider-agnostic and focuses on the conceptual decomposition of the system.
The following subchapters explain the implementation changes behind this decomposition and show how they improved portability, maintainability, and operational overhead across deployments.

\subsection{Codebase Modularization and Modernization} \label{subsec:codebase-modularization-and-modernization}
The revised implementation restructures the codebase around explicit modular boundaries.
The goal is to keep infrastructure concerns, application logic, and operational tasks separate so that changes remain local and easier to reason about.
This approach aligns with the fundamental principles of modularity.
Parnas~\cite{parnas_criteria_1972} argues that the effectiveness of modularization depends on the system's decomposition, highlighting that good decomposition enhances flexibility and comprehensibility by concealing design decisions within module boundaries.
Recent work on modular monolith architectures reports similar maintainability and reduced operational overhead benefits in modern cloud environments~\cite{al-qoran_modular_2025}.

The repository is organized as a monorepo with the top-level directories \path{apps/} and \path{packages/}.
The \path{apps/} directory contains the runnable units that are deployed independently, namely the main Nuxt web application and the two scheduled cleanup jobs.
The \path{packages/} directory contains reusable modules such as the provisioner, session, evaluator, database, mailer, and environment configuration packages.
These modules encapsulate specific functionality behind stable interfaces, which allows the runnable applications to remain comparatively thin orchestration layers.

This restructure also supports the architectural goal of decoupling maintenance from the main interactive learning workflow.
Both scheduled jobs in \path{apps/} reuse the same core modules in \path{packages/} as the main Nuxt application, but run as separate executables on a schedule.
This avoids keeping the main application running indefinitely solely to perform cleanup and enables deployments where the interactive application can be scaled down during idle periods while maintenance continues to enforce lifecycle and retention policies.

The modernization effort also included updating core frameworks and supporting dependencies to currently supported stable versions where feasible.
One example was the migration of the main application from Nuxt 3 to Nuxt 4, which addressed the outdated dependency baseline identified in subchapter~\ref{subsubsec:outdated-dependencies}.
In this way, dependency updates were treated as part of the broader codebase modernization effort rather than as isolated package-management tasks.

Clearer modular boundaries also improve validation, readability, and maintainability by enabling more localized reasoning and testing.
Structuring the codebase into reusable packages and independently runnable applications makes it easier to extend the system without destabilizing the overall workflow of the platform.
The next subchapter builds on this structure by describing the provisioning abstraction and the lifecycle management approach implemented behind the provisioning interface.

\subsection{Container Provisioning and Lifecycle Management} \label{subsec:container-provisioning-and-lifecycle-management}
The updated system introduces a provider-agnostic provisioning layer responsible for the creation, maintenance, and deletion of the user-specific Linux containers required for interactive terminal sessions.
This is a central change from the baseline, as it encapsulates provider-specific runtime logic behind a stable API, allowing the same user-facing workflow to operate across different execution backends.

\subsubsection{Provisioner Interface and Shared Abstraction} \label{subsubsec:provisioner-interface-and-shared-abstraction}
To decouple the core application from provider-specific details, the system defines an abstract provisioning interface that all backends must implement.
Upon application startup, a factory function evaluates the environment configuration and instantiates the appropriate provisioner implementation.
This approach ensures that the application relies solely on a stable API, while provider-specific logic is contained within the implementation classes.

At a high level, the provisioner supports four operations.
It can list currently active user containers, ensure a user container exists and create one if it is not present, update the container expiration time while the user remains active, and delete the container once it is no longer required.
Together, these four operations cover the full container lifecycle required by the application while leaving provider-specific startup, polling, metadata handling, and secret management inside the implementation classes.
The provisioner interface is shown in the code snippet below.

\begin{minted}{typescript}
export interface Provisioner {
  listContainers: () => Promise<ContainerInfo[]>;
  ensureContainerExists: (userId: string) => Promise<ConnectionInfo>;
  updateContainerExpiration: (userId: string) => Promise<void>;
  deleteContainer: (userId: string) => Promise<void>;
}
\end{minted}

The provisioning interface is implemented by classes specific to each provider, while shared operational concerns are centralized in a common base class, \texttt{AbstractProvisioner}.
In this base class, each provisioning operation is wrapped with a concurrency limiter to cap simultaneous operations against provider APIs, a per-user lock to serialize operations per user, and retry logic to handle transient failures.
For concurrency limiting and retry handling, the p-limit and p-retry packages are used.
The following code snippet demonstrates this wrapping for one of the interface functions, illustrating how the public method enforces shared guarantees while delegating provider-specific work to the underlying implementation method.

\begin{minted}{typescript}
async ensureContainerExists(userId: string): Promise<ConnectionInfo> {
  return this.withConcurrencyLimit(() =>
    this.withLock(userId, () =>
      this.withRetry(() => this.ensureContainerExistsImpl(userId)))
  );
}

protected abstract ensureContainerExistsImpl(userId: string):
  Promise<ConnectionInfo>;
\end{minted}

In practice, these wrappers are important under classroom-scale usage patterns and bursty traffic, such as page refreshes and repeated session starts.
Per-user serialization prevents race conditions where concurrent requests attempt to create or delete the same user's container.
Concurrency limiting reduces the likelihood of overwhelming provider APIs during spikes, and retry logic improves robustness against temporary network failures or throttling.
Additionally, the base class provides shared utilities for lifecycle management, including computing expiration timestamps, applying consistent metadata tags or labels, and generating fresh SSH key pairs for new containers, ensuring consistent behavior across providers.

\subsubsection{Lifecycle Management and Provider Implementations} \label{subsubsec:lifecycle-management-and-provider-implementations}
The provisioner abstraction presents a provider-agnostic API\@.
For example, \texttt{listContainers()} returns a list of user identifiers and expiration timestamps for all managed containers regardless of the underlying runtime.
Likewise, \texttt{ensureContainerExists(userId)} guarantees that the system can obtain connection information for a ready interactive container even though the internal handling differs per provider.
Provider implementations therefore encapsulate the relevant state transitions, such as creating a missing container, waiting while a container is starting, and refreshing expiration metadata when a container is already running.

Currently, three execution backends are supported.
These are Kubernetes, Azure Container Instances (ACI), and Amazon ECS on AWS Fargate.
All implementations follow the same lifecycle model by attaching common metadata to managed resources, such as the user identifier and expiration time, enabling consistent enumeration and reclamation of inactive containers.
The deployment setup and provider-specific configuration are described in chapter~\ref{sec:deployment-environments-and-infrastructure-setup}.

Lifecycle management is treated as a first-class operational concern to prevent resource leakage and control costs.
User containers are intentionally time-bound and carry an inactivity-based expiration timestamp that is extended during activity via \texttt{updateContainerExpiration()}.
Enforcement of expiration is separated from the main web application via the scheduled provisioner cleanup job described in subchapter~\ref{subsec:architecture-overview}.
At a high level, the cleanup job periodically lists managed containers, identifies those whose expiration time is in the past, and deletes them through the same interface.
This separation ensures that expired containers are reclaimed even if the main application is scaled down during idle periods, aligning the operational footprint more closely with actual usage.

Although this subchapter focuses on the provisioner module because it is the most complex, the same interface-driven design pattern is applied across other internal packages.
For example, the session module encapsulates the WebSocket and SSH bridging responsibilities behind a stable API, and the evaluator package isolates output buffering and regex-based validation logic.
Even where only a single implementation exists today, these boundaries keep responsibilities explicit and reduce the likelihood that infrastructure or operational concerns leak into the user-facing application.

\subsection{Authentication and Authorization} \label{subsec:authentication-and-authorization}
In the authentication layer, the revised system replaces the Lucia package with an actively maintained solution that supports session management and single sign-on (SSO) based on OAuth 2.0 and OpenID Connect (OIDC).
This change removes the dependency on the deprecated library discussed in subchapter~\ref{subsubsec:deprecated-authentication-library} and provides a more maintainable basis for secure access control.

\subsubsection{Authentication Design and Implementation} \label{subsubsec:authentication-design-and-implementation}
Authentication is essential for this learning platform to identify users and ensure continuity of sessions for individual Linux container environments.
User accounts enable the system to associate and store task results for each student and to limit access to authorized users.
This approach preserves progress across sessions and reduces the risk of misuse or excessive load by requiring authenticated access before provisioning cost-incurring resources.

Given Lucia's deprecation, continuing to rely on it was not suitable for a production system.
As an alternative, the updated system adopts Better Auth, an authentication library that provides built-in support for common authentication workflows and simplifies long-term maintenance~\cite{noauthor_better_nodate}.
In the revised system, the authentication layer supports email-based login, session management, and OIDC integration for SSO~\cite{noauthor_better_nodate}.
In addition, the revised system implements an email domain allowlist to restrict registration and login to approved domains, for example, \texttt{ut.ee}.
This helps ensure the platform is used only by the intended audience.

Compared to Lucia, Better Auth reduces the amount of custom authentication code by providing standardized handlers and integrations for the required workflows.
This decreases the likelihood of implementation errors, makes the authentication layer easier to evolve alongside the rest of the stack, and provides a standards-based basis for SSO onboarding.

\subsubsection{OpenID Connect Based Single Sign On} \label{subsubsec:openid-connect-based-single-sign-on}
OpenID Connect (OIDC) is an authentication protocol built on top of the OAuth 2.0 framework.
Fett et al.~\cite{fett_comprehensive_2016} describe OAuth 2.0 as a widely deployed authorization framework and as the foundation for OpenID Connect.
Fett et al.~\cite{fett_web_2017} further explain that OIDC allows a relying party to delegate authentication to an identity provider and to verify the result using a cryptographically signed id token.

At a high level, the platform uses the OIDC authorization code flow.
As described by Fett et al.~\cite{fett_web_2017}, the user is redirected from the application to the identity provider, authenticates there, and is then redirected back with an authorization code that is exchanged for tokens and used to establish a session.

In the current implementation, this flow can also use PKCE (Proof Key for Code Exchange), which adds a verification step during token exchange and improves resistance to authorization code interception.
The web application currently supports both Microsoft Entra ID, which was used during testing, and Keycloak, which is used for university integration, as OIDC providers through environment variable configuration.
After successful token validation, the authenticated external identity is mapped to the platform's internal user and session model, which is then used for progress tracking and authorization decisions.
This gives the platform a direct path to university identity integration and reduces onboarding friction in deployments where institutional login is available.

A standards-based OIDC integration improves interoperability and maintainability compared with provider-specific or custom authentication logic.
Fett et al.~\cite{fett_comprehensive_2016} show that secure OAuth-based systems depend on correct flow handling and validation under recommended best practices, while Fett et al.~\cite{fett_web_2017} provide the corresponding OIDC-specific analysis.
In the context of this platform, OIDC therefore provides a suitable and extensible basis for single sign-on support.

\subsection{Improved Localization and Task Editor} \label{subsec:improved-localization-and-task-editor}
The revised system improves both the internal localization model and the administrative workflow for managing multilingual learning content.
Although the baseline platform already supported multiple interface languages, the updated implementation extends this to the content model itself.
This was achieved by redesigning topics and tasks so that language-independent structure is separated from language-specific text, and by introducing an editor that supports both single-language and dual-language authoring.

\subsubsection{Localization Model} \label{subsubsec:localization-model}
A significant improvement in the revised implementation is the use of separate translation tables for topics and tasks.
In the previous system, equivalent learning content had to be duplicated per language.
In the updated schema, the topics and tasks tables store the stable logical structure of the educational content, including identifiers, ordering, and task evaluation configurations.
Additionally, the locale-specific textual fields are stored separately in \texttt{topic\_translations} and \texttt{task\_translations} tables.
Consequently, titles, descriptions, and hints are no longer tied directly to the structural entity but are instead attached as translations for each locale.

This design eliminates the need to duplicate topics or tasks across multiple languages.
It also ensures that the logical components of a task, such as its order, regular expression, or monitored path, are shared across all language variants, which reduces the risk of structural divergence between the Estonian and English versions over time.
Additionally, the uniqueness constraints on entity and locale combinations guarantee that each topic or task is limited to a single translation per language, thereby maintaining a predictable and consistent multilingual content model.

An additional benefit of this structure is that user progress is maintained independently of language.
Task completion is recorded using stable task and topic identifiers rather than specific translation entries.
Consequently, users can switch between supported interface locales without disrupting continuity, while still seeing the same logical exercise sequence in their chosen language.
This enhances the robustness of the localization model compared to a duplication-based approach, where each language version would function as a separate exercise set.

\subsubsection{Task Editor} \label{subsubsec:task-editor}
To support the revised localization model, the administrative task editor was redesigned to manage both single-language and dual-language content entries.
Instead of storing a separate configuration flag for editor mode, the implementation determines the mode directly from the available topic translations.
If only one locale contains content, the editor functions in single-language mode for that locale.
If both Estonian and English titles are available, the editor transitions to dual-language mode.
If neither locale is yet populated, the editor falls back to a default locale, allowing content creation to begin in a predictable way.

When bilingual content is required, the same editor supports both language variants within the same interface for the same underlying topic or task entity.

The editor also provides lightweight completeness feedback through status indicators shown in the outline and action area.
Tasks or topics with all required values present are marked with the indicator ``OK'', whereas entries with missing mandatory fields are marked as ``MISSING''.
This allows administrators to identify incomplete translations or task definitions without opening every item individually, which is particularly useful when editing larger bilingual exercise sets.
As a result, the redesign improves the editorial workflow by making it easier to maintain parity between language variants while also reducing duplication in content management.
