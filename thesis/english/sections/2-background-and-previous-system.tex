\section{Background and Previous System} \label{sec:background-and-previous-system}
This chapter establishes the baseline for the thesis.
It first introduces a browser-based Linux command-line practice environment and summarizes prior development efforts, starting from the initial prototype and continuing with the later bachelor's thesis implementation.
It then outlines the architecture and Azure deployment of the previous system that serves as the starting point for this master's thesis.
Finally, the chapter identifies the main limitations and outlines the improvement opportunities that guide the design and evaluation in later chapters.

\subsection{Motivation and Prior Work} \label{subsec:motivation-and-prior-work}
Linux command-line knowledge remains an important skill in many computing roles, particularly in system administration and automation.
Consequently, hands-on command-line practice is a relevant educational goal.
Earlier development work at the University of Tartu explored how such practice can be supported using a browser-based platform, beginning with an initial prototype and progressing to a more stable cloud-hosted implementation.

\subsubsection{The Importance of the Linux Command Line} \label{subsubsec:the-importance-of-the-linux-command-line}
Proficiency in command-line operations is a valuable skill due to the widespread use of Linux-based systems in modern computing infrastructure, which are often operated and maintained through terminal-centric workflows.
An observable indicator of Linux's prevalence in production computing environments is provided by high-performance computing.
According to TOP500 statistics, Linux is the operating system family used by all TOP500-ranked supercomputers~\cite{noauthor_operating_nodate}.
In such environments, administration, troubleshooting, and automation typically depend on command-line tools and remote terminal access instead of graphical interfaces.

This view aligns with academic perspectives on skills and learning.
Švábenský et al.~\cite{svabensky_toolset_2021} characterize the Linux command-line interface as common both in higher education computing environments and in software development practices.
They also advocate for studying command-line interactions to enhance comprehension and facilitate hands-on learning.
Likewise, Bailey and Zilles~\cite{bailey_uassign_2019} treat terminal skills as a topic worth teaching and assessing through authentic terminal actions, and they present an approach for interactive, auto-graded terminal assignments.

At the University of Tartu, Linux and its command line are treated as core topics in systems-oriented teaching.
For example, Linux command-line work plays a central role in the ``Operating Systems'' and ``Computer Security'' courses.
This instructional need motivated the development of a browser-based learning environment in two main iterations.
The first was a proof-of-concept prototype described by Halapuu~\cite{halapuu_eestikeelse_nodate}, and the second was a cloud-hosted implementation presented in Eistre's bachelor's thesis~\cite{eistre_veebipohise_nodate}.
These two iterations are summarized in the following subchapters.

\subsubsection{Initial Proof-of-Concept} \label{subsubsec:initial-proof-of-concept}
Halapuu~\cite{halapuu_eestikeelse_nodate} presents the first iteration of the browser-based Linux command-line learning environment.
In Halapuu's thesis, the prototype provides students with a web terminal connected to an isolated Linux environment, enabling interactive command execution and task-based learning within the browser.
The prototype also includes an automatic task-checking approach based on observable outcomes such as command output and filesystem changes, which allowed students to receive immediate feedback and progress through tasks.

Halapuu~\cite{halapuu_eestikeelse_nodate} further reports that the proof-of-concept application was deployed inside the University's internal network and run on the supervisor's university laptop.
The platform was tested in the ``Computer Security'' course context with over one hundred participants.
Based on the reported evaluation, he concludes that the overall approach was viable for course use, while also identifying improvement opportunities related to stability, security, and maintainability.

\subsubsection{Cloud-Hosted Implementation} \label{subsubsec:cloud-hosted-implementation}
After the initial proof-of-concept had been used in the supervisor's courses for two years, Eistre~\cite{eistre_veebipohise_nodate} advanced the system into a cloud-hosted platform designed for sustained course use.
In Eistre's thesis, the work addresses limitations identified during previous use and sets goals such as improving stability and maintainability, supporting both Estonian and English, and deploying the system in a more robust cloud environment.

Eistre~\cite{eistre_veebipohise_nodate} reports that the resulting environment was evaluated in the context of the ``Computer Security'' course in February 2024.
During this period, the platform was used by 382 participants.
He concludes that the overall outcome was successful, while also noting minor issues observed during the session, which informed further improvements for continued production use.

The following subchapter outlines the architecture and cloud deployment of the baseline system, which serves as the starting point for the work presented in this thesis.

\subsection{Architecture and Deployment of the Previous System} \label{subsec:architecture-and-deployment-of-the-previous-system}
This subchapter outlines the baseline system developed in Eistre's thesis~\cite{eistre_veebipohise_nodate}, which serves as the technical starting point for this master's thesis.
It first describes the application's internal architecture and operational flow, focusing on its main components and technological choices.
It then summarizes the Azure-based deployment architecture, including the services used to host the application and provision user-specific container environments.

\subsubsection{System Architecture} \label{subsubsec:system-architecture}
Eistre~\cite{eistre_veebipohise_nodate} outlines the baseline system as a production-oriented web-based Linux command-line learning environment redesigned for cloud deployment and sustained course use.
The implementation was largely rewritten in TypeScript using Node.js and the Nuxt 3 framework, which provided a structured basis for both the user interface and server-side logic.

Eistre~\cite{eistre_veebipohise_nodate} structures the environment around four main components: a client-side user interface, a server-side application interface (API), a MySQL database, and a Kubernetes cluster that provisions isolated Linux environments.
At runtime, the browser-based terminal communicates with the API layer, which brokers interaction with a user-specific Linux container in Kubernetes and records user progress and task state in MySQL\@.
The client-side interface is developed using Vue.js with Nuxt and supports features like user registration and authentication, topic navigation, task solving, and administrative views for managing topics and tasks.
The interface also supports both English and Estonian, as well as theme switching.

Eistre~\cite{eistre_veebipohise_nodate} conceptualizes the server-side application as the central coordination layer of the platform.
It manages authentication and sessions, exposes API endpoints consumed by the user interface, and orchestrates operations that require persistence or interaction with the Linux execution environment.
In particular, the backend provides user-specific Linux environments, manages their lifecycle, and brokers terminal interaction between the browser and the container runtime.
It also determines task completion based on predefined criteria and records progress updates when completion conditions are satisfied.
This positions the backend as the primary integration point among the user interface, the database, and the container runtime.

Eistre~\cite{eistre_veebipohise_nodate} reports that the baseline system uses a MySQL database to maintain persistent state, including user accounts, topics and their underlying tasks, and user progress.
Persisting these elements supports continuity across sessions and enables centralized management of task content without redeployment.
The database access layer is implemented using an Object-Relational Mapping (ORM) layer, providing typed access to relational entities and simplifying the handling of relationships between users, topics, and tasks.

Eistre~\cite{eistre_veebipohise_nodate} further explains that the learning environment uses containerization to provide isolated Linux execution environments and employs Kubernetes to manage the lifecycle of per-user containers (pods).
When a user initiates a terminal session, the system allocates a dedicated Kubernetes namespace and provisions an isolated Linux environment for that user, enabling repeatable task execution while preventing cross-user interference.
Since these environments consume compute resources, the architecture incorporates automated cleanup procedures that terminate inactive user environments after a configured idle period, reclaiming resources and keeping the cluster load manageable during course usage.
He also notes that the backend interacts with the cluster programmatically using the Kubernetes JavaScript Client library.

This foundational architecture serves as the initial framework for the cost and deployment optimizations that are subsequently introduced in this thesis.

\subsubsection{Cloud Deployment on Azure} \label{subsubsec:cloud-deployment-on-azure}
Eistre~\cite{eistre_veebipohise_nodate} reports that the baseline system was deployed on Microsoft Azure.
Azure was chosen in part because it offers educational credits to students, which made it a cost-effective choice for an academic project.
In his thesis, the application was containerized and deployed using Azure's managed container and Kubernetes offerings, specifically Azure Container Apps (ACA) and Azure Kubernetes Service (AKS).
Figure~\ref{fig:azure-deployment-overview} provides a deployment-level overview of the baseline environment and the Azure services created for it.

In this design, the Nuxt application was packaged into a container image and run as a single Azure Container App instance, with fixed resource reservations, including 0.5 vCPU and 1 GB of memory, in the Sweden Central region~\cite{eistre_veebipohise_nodate}.
According to Microsoft's documentation, Azure Container Apps is a serverless container hosting service that allows applications to run without managing the underlying infrastructure~\cite{noauthor_azure_nodate}.
The service handled execution of the web application and acted as the primary entry point for users by exposing the user interface and backend API endpoints.

While Azure Container Apps hosts the web application itself, Eistre~\cite{eistre_veebipohise_nodate} notes that the interactive Linux environments required for the terminal tasks are provisioned separately on Azure Kubernetes Service (AKS).
According to Microsoft's documentation, AKS is a managed Kubernetes service that offloads control-plane management, simplifies cluster operations, and offers scalability~\cite{noauthor_what_nodate}.
Eistre~\cite{eistre_veebipohise_nodate} points out that the cluster was configured using B2s virtual machines (2 vCPU, 4 GB memory, 128 GB storage) and that, although the application ran in Sweden Central, the cluster was deployed in Norway East due to limitations of the Azure student subscription.

A central motivation for the Azure-based deployment was cost-aware operation.
Eistre~\cite{eistre_veebipohise_nodate} highlights that continuously running a Kubernetes cluster can incur significant baseline cost even when there is no active classroom usage.
To address this, the baseline system implements an AKS control mechanism where the application can start and stop the cluster based on demand.
When a user initiates a session while the cluster is stopped, the platform triggers cluster startup and presents a waiting state until the cluster becomes available.
Conversely, he also notes that when the system is idle for a configured period, the cluster can be stopped to avoid paying for unused compute capacity.

Eistre~\cite{eistre_veebipohise_nodate} also describes a small set of supporting Azure services created as part of the baseline deployment in addition to compute and container orchestration.
These include Azure Communication Services for email delivery, which is used for account verification, a storage account, which is used to persist private key pairs across restarts, and a MySQL-backed database for persisting users, topics, tasks, and progress.
He also notes that the database was initially hosted on PlanetScale and later migrated to Azure's managed MySQL offering after PlanetScale discontinued the free tier.
Figure~\ref{fig:azure-deployment-overview} summarizes the high-level dependencies between these services and the end-to-end flow from the client to the provisioned environment.

This Azure deployment model forms the baseline operational environment for this thesis, which focuses on improving cost efficiency and deployment robustness while preserving the functional behavior of the system described in this subchapter.

\subsection{Limitations and Improvement Opportunities} \label{subsec:limitations-and-improvement-opportunities}

While the cloud-enabled platform developed in Eistre's bachelor's thesis~\cite{eistre_veebipohise_nodate} represented a significant advance over the initial prototype, it also exposed several limitations and challenges.
This subchapter summarizes the key issues in the baseline system and motivates the concrete improvements explored in later chapters of this thesis.

\subsubsection{Cost Factor} \label{subsubsec:cost-factor}
A significant limitation of the baseline system was its cost structure in the cloud-hosted environment.
Figure~\ref{fig:azure-cost-july-2025} illustrates the Azure cost breakdown for July 2025, a period of minimal usage.
In July, the deployment incurred a total monthly expense of €52.88.
The majority of these costs were attributed to fixed, always-on resources, such as the Azure Database for MySQL (€26.90), the AKS load balancer (€13.62), and Azure Container Apps hosting the application (€8.92).
Consequently, maintaining the platform’s readiness for use resulted in a baseline expenditure exceeding €50 per month, even with minimal or no user activity.

Although the baseline system had already implemented demand-based cluster start and stop mechanisms, the figures show that idle costs remained substantial, and peak usage continued to result in significant compute and supporting service charges.
Therefore, opportunities for improvement include minimizing the always-on footprint to bring idle costs closer to zero, as well as exploring alternative deployment strategies that are more cost-effective for intermittent workloads, while maintaining the functional requirements of the baseline interactive per-user Linux environments.

\subsubsection{Deprecated Authentication Library} \label{subsubsec:deprecated-authentication-library}
The previous version of the learning environment relied on the Lucia authentication library to implement core authentication functionality, including user registration, login, and session management~\cite{eistre_veebipohise_nodate}.
According to the Lucia maintainer, the project would be deprecated by March 2025 and would transition away from being a maintained, general-purpose authentication library~\cite{noauthor_fresh_nodate}.
Consequently, Lucia is now positioned primarily as an educational resource for learning about authentication implementation rather than a long-term dependency for production systems~\cite{noauthor_fresh_nodate}.
This shift indicates that Lucia is no longer a dependable long-term solution for the platform’s authentication layer.

This deprecation poses a significant risk to the platform’s maintainability.
Continued dependence on an unsupported library may result in delayed or nonexistent security updates, increased friction when upgrading adjacent frameworks, and elevated long-term maintenance costs.
To mitigate these risks, the system should be updated to integrate an authentication library or service that is actively maintained and widely adopted, while preserving the baseline requirements for user verification and session management.
This modification will improve the long-term maintainability and security of the platform’s authentication layer.

\subsubsection{Outdated Dependencies} \label{subsubsec:outdated-dependencies}
A further limitation of the baseline system is its reliance on dependencies that inevitably become outdated over time, creating both maintainability and security risks.
Prior research indicates that delayed library updates are common in web ecosystems and are linked to exposure to known vulnerabilities.
Lauinger et al.~\cite{lauinger_thou_2017} discovered that a significant proportion of websites contain at least one JavaScript library version with a known vulnerability, with the time lag behind the latest release often measured in years.
Furthermore, Pashchenko et al.~\cite{pashchenko_qualitative_2020} report that decisions regarding dependency updates frequently involve trade-offs between security, compatibility, and engineering effort, which can result in postponed updates even when the associated risks are recognized.

These risks are particularly relevant for framework-level dependencies that form the core of the application stack.
The baseline system was developed using Nuxt 3~\cite{eistre_veebipohise_nodate}, whereas newer stable major releases, such as Nuxt 4, are now available~\cite{noauthor_roadmap_nodate}.
In addition to lifecycle considerations, older framework versions may contain publicly disclosed vulnerabilities.
For instance, CVE-2025-27415 identifies a cache-poisoning issue affecting Nuxt versions prior to 3.16.0 in specific CDN (content delivery network) configurations, which was fixed in Nuxt 3.16.0~\cite{noauthor_nvd_nodate}.
This example illustrates that dependency versions can directly influence the security and reliability of the system.

Taken together, these findings motivate dependency modernization as a vital improvement objective in this thesis.
To reduce exposure to known vulnerabilities and to maintain compatibility with current runtimes and tooling, the implementation should update core frameworks and supporting packages to currently supported, stable versions as part of the modernization effort.

\subsubsection{General Improvements} \label{subsubsec:general-improvements}
In addition to the limitations previously described, several further opportunities for improvement were identified in the design of the baseline system.
The baseline demonstrated an operational scalability limitation under classroom-scale load.
The thesis supervisor noted that in some instances students were unable to access an interactive shell session because the AKS node pool could not scale due to Azure quota constraints.

Second, the content model did not natively support localization for topics and tasks.
Although the user interface supported multiple languages, equivalent learning content had to be duplicated per locale, increasing maintenance overhead and creating a risk of inconsistencies when content changed.
A key improvement is to store each topic and task once and attach language-specific translations in the database, enabling consistent updates while still presenting localized content to users.

Finally, the baseline authentication approach can be improved from a usability perspective by integrating the University’s single sign-on capabilities.
Enabling institutional login would simplify onboarding and align access control with existing university identity management, thereby improving the overall user experience.

\subsubsection{Infrastructure as Code} \label{subsubsec:infrastructure-as-code}
An additional limitation of the baseline system was the high setup overhead needed to replicate the environment.
Deploying the platform required cloud knowledge and coordination of several prerequisites, such as managed database provisioning, container hosting configuration, and Kubernetes cluster setup.
In practice, this involved following a comprehensive, step-by-step deployment procedure, which is challenging to replicate consistently across environments.
This type of manual, interactive infrastructure configuration is widely recognized as slow and prone to mistakes, especially as system complexity grows~\cite{guerriero_adoption_2019}.

This limitation can be addressed through DevOps practices that emphasize automation and repeatability across development and operations.
Guerriero et al.~\cite{guerriero_adoption_2019} describe DevOps as practices that combine software development and IT operations, and they position Infrastructure as Code as a DevOps tactic where infrastructure is managed through reusable definition files rather than interactive configuration.
Additionally, Bali and Walia~\cite{bali_enhancing_2023} note that IaC emerged to mitigate configuration drift and to reduce reliance on individually configured, non-reproducible environments, and that representing infrastructure state as code improves repeatability while reducing manual, error-prone operations.
Together, these perspectives motivate adopting IaC for platform deployment as a means to reduce setup friction, improve reproducibility, and lower the risk of human error.
