\section{Deployment Environments and Infrastructure Setup} \label{sec:deployment-environments-and-infrastructure-setup}
This chapter describes how the revised Linux command-line learning environment was deployed in three target environments.
It first outlines the deployment goals and the rationale for selecting University of Tartu HPC (UT HPC), Microsoft Azure, and Amazon Web Services as target platforms.
It then describes the reasoning behind each deployment together with the main infrastructure components, deployment approach, and operational trade-offs.
Together, these deployments provide the practical basis for the cross-platform evaluation presented in the next chapter.

\subsection{Deployment Goals and Platform Selection} \label{subsec:deployment-goals-and-platform-selection}
The revised Linux command-line learning environment was designed to run across multiple deployment environments while preserving the same core application functionality.
The goal of this multi-platform approach was not only portability but also to enable a structured comparison between different hosting models and to identify the most suitable long-term deployment environment for continued use.
For that reason, this chapter discusses deployment separately from the application architecture.
Chapter~\ref{sec:revised-architecture-and-implementation} focused on the internal design and implementation of the system, whereas this chapter focuses on how that design was deployed in different environments and what trade-offs those environments introduced.

Across all target environments, the deployment design was guided by several shared goals.
First, the environment had to support isolated Linux-based execution environments for individual users, as this is central to the teaching model of the system.
Second, the deployment process needed to become reproducible and maintainable, replacing the more manual setup process of the earlier implementation with a clearer and more structured approach.
Third, the system needed reliable persistence and integration between the application, database, and supporting services.
Finally, because one of the main goals of this thesis is cost reduction, deployment choices also had to account for operational overhead and cost efficiency.

Based on these goals, University of Tartu HPC (UT HPC) was selected as one of the deployment targets.
This platform was particularly relevant because the original application model was already based on containers and Kubernetes, making the university-managed Kubernetes service a natural continuation of that design.
In addition, UT HPC provides a managed Kubernetes environment inside the University of Tartu network, where users are given access to dedicated namespaces rather than operating the cluster itself.
This made it a meaningful candidate both as a technically compatible platform and as a realistic university-hosted deployment model.

Microsoft Azure and Amazon Web Services were selected as the two public cloud platforms for this comparison.
Azure was included because the previous thesis implementation already used Azure, making it the most direct baseline for modernization and cost comparison.
AWS was included to extend the comparison beyond a single provider and to evaluate how well the redesigned system could be deployed in a second major cloud environment.
Together, these three platforms provide a basis for comparing institutional hosting and public cloud deployment models in a consistent manner while keeping the application architecture itself largely unchanged.

\subsection{Deployment Automation and Infrastructure as Code} \label{subsec:deployment-automation-and-infrastructure-as-code}
Reproducible deployment was a shared design goal across all target environments.
However, the environments differed in what had to be automated.
In UT HPC, the cluster infrastructure already existed and was managed externally.
By contrast, the Azure and AWS deployments also required provisioning the surrounding cloud services needed to run the environment.
For that reason, different automation approaches were used.

For UT HPC, the application was deployed using Helm.
Helm is described by its maintainers as a package manager for Kubernetes that simplifies the deployment and lifecycle management of applications through charts~\cite{noauthor_helm_nodate}.
In Helm terminology, a chart is a structured package containing the files needed to describe an application's Kubernetes resources, including templates that are rendered into manifests using supplied configuration values~\cite{noauthor_charts_nodate}.
In this project, the Helm chart bundled the learning environment's Kubernetes resource templates while separating reusable defaults from environment-specific settings.
Helm supports this by allowing the baseline configuration from \path{values.yaml} to be overridden through user-provided values files or command-line parameters~\cite{noauthor_values_nodate}.
This made Helm suitable for the UT HPC deployment, where the main task was to install and configure namespace-scoped Kubernetes resources inside an already available managed cluster.

For Azure and AWS, the infrastructure was provisioned using Terraform.
According to the Terraform documentation, Terraform is an infrastructure-as-code (IaC) tool that allows infrastructure to be specified in configuration files and managed through a repeatable workflow~\cite{noauthor_what_nodate-1}.
Terraform supports this by providing input variables, which make it possible to supply configuration values separately from the core infrastructure code~\cite{noauthor_use_nodate}.
This made it possible to keep reusable infrastructure definitions while supplying deployment-specific values for each cloud environment, which improved deployment reproducibility.
Brief deployment instructions for these environments are provided in Appendix~\ref{app:deployment-instructions}.

\subsection{University of Tartu HPC} \label{subsec:university-of-tartu-hpc}
This subchapter presents the deployment of the revised platform in UT HPC\@.
In addition to serving as one of the target platforms in the cross-environment comparison, this deployment is relevant as a concrete example of institutional hosting.
The following subchapters first introduce UT HPC and explain the namespace access process, and then describe the deployed components and their operational considerations.

\subsubsection{UT HPC as a Deployment Environment} \label{subsubsec:ut-hpc-as-a-deployment-environment}
The University of Tartu HPC Centre provides multiple research- and teaching-oriented computing services, including a managed Kubernetes service for hosting containerized workloads.
According to the UT HPC service description, the Kubernetes cluster is deployed on bare-metal nodes within the University of Tartu network in a high-availability configuration, while users are granted managed, namespace-level access rather than responsibility for administering the underlying cluster infrastructure~\cite{center_kubernetes_nodate}.
In this model, projects are assigned dedicated namespaces, enabling application deployment and management within a namespace-scoped environment, while broader cluster administration remains outside the scope of ordinary user access~\cite{noauthor_quick_nodate}.
This made UT HPC relevant in this thesis both as a technically compatible deployment target for the Kubernetes-based platform and as a practical example of deployment in university-managed infrastructure.

\subsubsection{Access and Namespace Provisioning} \label{subsubsec:access-and-namespace-provisioning}
Access to the UT HPC Kubernetes cluster was obtained through the namespace request process described in the UT HPC documentation.
According to the UT HPC Kubernetes quick start guide, access can be requested either by submitting a Kubernetes namespace request or by contacting HPC support directly~\cite{noauthor_quick_nodate}.
As part of the request, the applicant can specify details such as the domain name, deployment lifetime, project repository, and whether the application requires a database~\cite{center_container_nodate}.
After the request has been processed, the user is notified by email and access is granted at the namespace level~\cite{noauthor_quick_nodate}.

As part of this thesis, the application was deployed into the namespace terminal-cs-ut-ee.
After the namespace had been provisioned, cluster access was configured using the kubeconfig file-based access model described in the UT HPC documentation~\cite{noauthor_quick_nodate}.
In addition to the namespace provisioning itself, several deployment-specific integrations were coordinated directly with UT HPC support by email.
These included exposing the application under the public hostname terminal.cs.ut.ee through the UT HPC ingress infrastructure~\cite{noauthor_ingress_nodate}, configuring single sign-on through an HPC-managed Keycloak instance, and configuring Simple Mail Transfer Protocol (SMTP) settings required for sending account verification emails.
UT HPC support also provided a monitoring dashboard for standard workload metrics, including pod CPU usage, memory usage, and the number of active student containers\footnote{Based on private communication with UT HPC support during the deployment process.}.

\subsubsection{Deployed Components and Operational Considerations} \label{subsubsec:deployed-components-and-operational-considerations-helm}
The application was installed in the terminal-cs-ut-ee namespace and made accessible to users via the public hostname terminal.cs.ut.ee.
Deployment into the namespace was performed using the project's Helm chart with environment-specific values for UT HPC, including ingress-related options and database configuration.
The Helm release deployed the main web application together with the Kubernetes resources required to run the platform in this target environment.
These included the application Deployment, the public-facing Service and Ingress, the database layer configured using the MariaDB\footnote{\url{https://mariadb.org/}} operator, cleanup CronJobs, and namespace-level resources such as service accounts, role bindings, and network policies.
The deployment remained fully contained within the designated namespace, while public access was provided through the UT HPC ingress infrastructure~\cite{noauthor_ingress_nodate}.
Figure~\ref{fig:ut-hpc-deployment-overview} presents the deployment of the learning environment in UT HPC, showing the namespace-scoped application components, ingress path, database layer, and per-user containers.

\begin{figure}[ht]
  \centering
  \includegraphics[width=\textwidth]{figures/figure8-university-deployment}
  \caption{Deployment architecture of the learning environment in UT HPC.}
  \label{fig:ut-hpc-deployment-overview}
\end{figure}

Public access was implemented through a Kubernetes Ingress resource.
UT HPC documentation identifies ingress as a supported method for publishing HTTP applications to the public internet through its managed proxy infrastructure, while routing, DNS records, and TLS certificates were coordinated with HPC support~\cite{noauthor_ingress_nodate}.
Within the namespace, the web application was deployed as a standard Kubernetes Deployment behind a Service, with optional autoscaling available through chart configuration\@.
Additionally, the web application was given explicit resource requests of 250~mCPU and 512~MiB of memory, with limits of 500~mCPU and 1~GiB\@.
Configuration was kept separate from the application code by storing non-sensitive settings in ConfigMaps and providing sensitive values as Secrets.
Service accounts and access control resources were used to define the permissions required by the application and its supporting workloads.
Network policies were additionally used to restrict communication paths between the main application, the database, and the per-user execution environments.

The database layer was deployed using MariaDB, one of the supported database options in UT HPC~\cite{noauthor_databases_nodate}.
This was also appropriate for the project because MariaDB documentation describes it as having historically functioned as a drop-in replacement for equivalent MySQL versions, making it a compatible continuation of the earlier MySQL-based setup~\cite{noauthor_mariadb_2026}.
The operator-managed database workload requested 300~mCPU and 256~MiB of memory, had a memory limit of 1~GiB, and used 1~GiB of persistent storage.
In addition to the main web application and database components, the chart deployed periodic cleanup jobs to remove expired session resources and other stale state.
These cleanup CronJobs requested 50~mCPU and 128~MiB of memory, with limits of 100~mCPU and 256~MiB, which kept their baseline footprint small while reserving enough capacity for scheduled maintenance tasks.
Per-user execution environments were integrated into the same namespace-scoped deployment model, allowing interactive student containers to be created and managed alongside the main environment while remaining within the security and networking boundaries defined for the namespace.
Each per-user terminal container requested 100~mCPU and 64~MiB of memory, with limits of 250~mCPU and 128~MiB\@.

From an operational perspective, the deployment in UT HPC reduced the amount of infrastructure that had to be managed directly within the scope of the project.
The Kubernetes cluster and ingress infrastructure were provided by UT HPC, while the responsibility for the application deployment remained focused on namespace-scoped resources such as the Helm release, database setup, and supporting workloads.
At the same time, some parts of the deployment, including public exposure, single sign-on, and SMTP integration, depended on coordination with UT HPC support.
This made UT HPC a realistic university-hosted option for long-term course use.
It reduced direct infrastructure ownership and preserved Kubernetes compatibility, but it also introduced a stronger dependency on institutional support for parts of the operational setup.

\subsection{Microsoft Azure} \label{subsec:microsoft-azure}
This subchapter discusses the deployment of the revised environment on Microsoft Azure.
Azure was retained as a target environment because the baseline implementation described in subchapter~\ref{subsubsec:cloud-deployment-on-azure} was already hosted there, making it the most direct platform for modernization and comparison.
While the deployment in UT HPC shows how the environment can be hosted in institutional Kubernetes infrastructure, the Azure deployment represents the public cloud modernization path for the baseline system.

\subsubsection{Azure as a Deployment Environment} \label{subsubsec:azure-as-a-deployment-environment}
Microsoft Azure was selected as a deployment target because it provides the clearest continuation of the baseline system and therefore allows the revised design to be compared with the previous Azure-based deployment model.
As described in subchapter~\ref{subsubsec:cloud-deployment-on-azure}, the earlier implementation used Azure Container Apps (ACA) for the main web application and Azure Kubernetes Service (AKS) for the Linux execution environments~\cite{eistre_veebipohise_nodate}.
Although this architecture was functional, it still retained some of the cost and operational characteristics of cluster-based orchestration, since the execution environments relied on AKS worker nodes and related infrastructure even when user activity was low~\cite{eistre_veebipohise_nodate}.

For the revised implementation, Azure remained relevant, but the service choices were reconsidered.
The main goal was to preserve the same educational workflow while reducing idle infrastructure overhead and simplifying the operational model.
As a result, the revised Azure deployment replaces the cluster-centric execution model with a combination of managed and consumption-oriented services that better fit the environment's intermittent classroom usage pattern.
This made Azure suitable not only for continuity with the baseline system but also for evaluating whether a more lightweight Azure-native architecture could address the cost and maintenance limitations identified in chapter~\ref{sec:background-and-previous-system}.

\subsubsection{Deployed Components and Operational Considerations} \label{subsubsec:deployed-components-and-operational-considerations-azure}
The Azure environment was provisioned using Terraform configuration files.
In the resulting deployment, the revised Azure setup was centered around Azure Container Apps (ACA), which hosted the main web application, while the scheduled maintenance workloads were deployed as Azure Container Apps Jobs, which executed containerized tasks for a finite duration and then stopped~\cite{noauthor_azure_nodate,craigshoemaker_jobs_nodate}.
In this environment, the main Nuxt application ran as a containerized web workload configured with 0.5 CPU and 1~GiB of memory, while the database cleanup and provisioner cleanup functions ran as separate scheduled jobs, each configured with 0.25 CPU and 0.5~GiB of memory.
This matches the architectural separation described in subchapter~\ref{subsec:architecture-overview}, where periodic maintenance was moved out of the main application process.

\begin{figure}[ht]
  \centering
  \includegraphics[width=\textwidth]{figures/figure9-azure-deployment}
  \caption{Deployment architecture of the learning environment in Microsoft Azure.}
  \label{fig:azure-deployment-overview-revised}
\end{figure}

Figure~\ref{fig:azure-deployment-overview-revised} illustrates the revised Azure deployment.
Users access the environment through the public endpoint provisioned for the application by ACA, which then routes traffic to the main web application container.
When an interactive terminal session is requested, the application provisions a new user-specific Linux container through Azure Container Instances (ACI)~\cite{tomvcassidy_serverless_nodate}.
Each ACI session container was configured with 0.1~vCPU and 0.1~GB of memory.
ACI allows the environment to provide isolated containers for each user without the need to maintain virtual machines or cluster infrastructure~\cite{tomvcassidy_serverless_nodate}.
Compared with the baseline AKS model, this reduces both infrastructure and operational overhead because the application no longer needs to manage cluster runtime to reduce idle costs.
In this model, the runtime cost of the session containers is determined by the requested vCPU and memory resources together with the duration of the session~\cite{noauthor_pricing_nodate}.

The database layer was implemented using the same Azure Database for MySQL Flexible Server deployment model as in the baseline system, using the cheapest available configuration to minimize costs (B1ms burstable tier with 1 vCPU and 2 GB of memory)~\cite{noauthor_flexible_nodate,eistre_veebipohise_nodate}.
Unlike the baseline implementation, which relied on Azure Files\footnote{\url{https://azure.microsoft.com/en-us/products/storage/files}} to store per-session SSH keys, the revised deployment used Azure Key Vault\footnote{\url{https://azure.microsoft.com/en-us/products/key-vault}} for secret storage~\cite{msmbaldwin_azure_nodate,eistre_veebipohise_nodate}.
Similar to the baseline system, email delivery for account verification was handled through Azure Communication Services using an Azure-managed email domain~\cite{sroons_azure_nodate,bashan-git_overview_nodate,eistre_veebipohise_nodate}.
Regarding authentication, the Azure deployment did not integrate with the university-managed Keycloak service, as that setup was specific to UT HPC\@.
Accordingly, it followed the same pattern as the baseline Azure system, where users registered within the application and validated their accounts via email.
This approach kept the Azure deployment self-contained within the cloud environment.

From an operational perspective, the revised Azure deployment differs from the baseline primarily in its reduced dependence on cluster-based infrastructure.
The main web application and the scheduled jobs were deployed as managed container workloads, while session containers were provisioned on demand through ACI\@.
In addition, the ACA web application was configured with a minimum replica count of zero, which allowed the web layer to scale to zero during idle periods~\cite{craigshoemaker_scaling_nodate}.
This eliminated the need to stop and start a Kubernetes cluster to control idle costs and reduced startup-related operational overhead after periods of inactivity.
This design is better aligned with the environment's irregular usage pattern, where demand is concentrated around course sessions rather than spread uniformly over time.
Within the overall comparison, Azure therefore represents the public cloud deployment with the lowest idle footprint while reducing the always-on infrastructure burden of the earlier Azure-based design.

\subsection{Amazon Web Services} \label{subsec:amazon-web-services}
This subchapter describes the deployment of the revised environment on Amazon Web Services (AWS).
AWS was included as an additional public cloud platform to complement the Azure-based baseline and to evaluate how the redesigned system can be deployed in a second major cloud environment with a different service ecosystem.
The AWS deployment broadens the comparison by adding a second public cloud reference point for infrastructure automation, operational overhead, and cost drivers.

\subsubsection{AWS as a Deployment Environment} \label{subsubsec:aws-as-a-deployment-environment}
Amazon Web Services was selected as an additional public cloud deployment target to broaden the evaluation beyond a single cloud provider.
Similar to Azure, AWS provides a realistic environment for assessing how the redesigned environment can be deployed using managed cloud services rather than Kubernetes resources.
Unlike UT HPC, which represents institutional Kubernetes hosting, the AWS deployment reflects an alternative public cloud model based on managed services and on-demand resource usage.
This makes AWS a suitable comparison platform for evaluating whether the provider-agnostic architecture described in chapter~\ref{sec:revised-architecture-and-implementation} generalizes cleanly beyond Azure while keeping the core application requirements unchanged.

\subsubsection{Deployed Components and Operational Considerations} \label{subsubsec:deployed-components-and-operational-considerations-aws}
Like the Azure environment, the AWS environment was provisioned using Terraform configuration files.
In the resulting deployment, the AWS setup combined an always-on web hosting layer with on-demand and scheduled container workloads.
The main Nuxt web application was deployed to the Amazon Lightsail\footnote{\url{https://aws.amazon.com/lightsail/}} container service using the Nano plan, corresponding to 0.25 shared vCPU and 512~MB of RAM, while the database cleanup and provisioner cleanup workloads ran as scheduled ECS tasks on AWS Fargate~\cite{noauthor_deploy_nodate,noauthor_architect_nodate,noauthor_amazon_nodate}.
An alternative considered for the main web application was an ECS service behind an Application Load Balancer\footnote{\url{https://aws.amazon.com/elasticloadbalancing/application-load-balancer/}}~\cite{noauthor_use_nodate-1}.
Compared with the selected Lightsail container service setup, that design would introduce separate load-balancing infrastructure and a higher always-on cost baseline, so Lightsail container services were chosen as the simpler and more cost-effective option for the continuously running web layer~\cite{noauthor_elastic_nodate,noauthor_amazon_nodate}.

\begin{figure}[ht]
  \centering
  \includegraphics[width=\textwidth]{figures/figure10-aws-deployment}
  \caption{Deployment architecture of the learning environment in Amazon Web Services.}
  \label{fig:aws-deployment-overview-revised}
\end{figure}

Figure~\ref{fig:aws-deployment-overview-revised} illustrates the AWS deployment.
Users access the environment through the HTTPS endpoint of the Lightsail container service deployment, which routes requests to the main web application container~\cite{noauthor_create_nodate}.
When an interactive terminal session is requested, the application launches a new Linux container for the user on ECS using AWS Fargate.
Each session task was configured with 0.25 CPU and 512~MiB of memory.
In this design, session environments are created on demand as isolated tasks rather than kept running continuously.

For secret storage, the AWS deployment used AWS Secrets Manager\footnote{\url{https://aws.amazon.com/secrets-manager/}} to store the per-session SSH keys~\cite{noauthor_what_nodate-2}.
To support this workflow, the deployment also defined AWS IAM\footnote{\url{https://aws.amazon.com/iam/}} roles and credentials that allowed the web application to launch and manage ECS tasks and access the session secrets required for terminal provisioning.
For the database layer, data was stored in a managed MySQL database provided by Amazon Lightsail using the Micro plan, which provided 1~GB of memory, 1 core processor, and 40~GB of SSD storage~\cite{noauthor_select_nodate,noauthor_create_nodate-1}.
Regarding authentication, the AWS deployment did not integrate with the university-managed Keycloak service, as that setup was specific to UT HPC\@.

From an operational perspective, the AWS deployment used a Lightsail container service with the Nano plan for the always-on web layer~\cite{noauthor_amazon_nodate}.
The scheduled cleanup tasks used the same ECS task sizing as the session containers, namely 0.25 CPU and 512~MiB of memory.
As in the Azure deployment, the cost of terminal sessions depends on the requested resources and the runtime of the Fargate tasks~\cite{noauthor_aws_nodate}.
However, unlike the Azure ACA deployment, the Lightsail container service does not provide a comparable scale-to-zero configuration for the web layer.
Overall, the AWS deployment shows that the revised architecture can be transferred to another major cloud provider without requiring Kubernetes for session execution, although it does so with a continuously running web layer and a higher idle baseline than Azure.

Taken together, the three deployment environments represent distinct operational models.
UT HPC represents institutionally managed Kubernetes hosting.
Azure represents a public cloud deployment with the lowest idle footprint.
AWS represents a service-based design that avoids Kubernetes for session execution but retains a continuously running web layer.
The next chapter builds on these deployment cases by comparing their cost and operational consequences more directly.
