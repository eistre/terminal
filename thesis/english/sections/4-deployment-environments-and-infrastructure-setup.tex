\section{Deployment Environments and Infrastructure Setup} \label{sec:deployment-environments-and-infrastructure-setup}
This chapter describes the deployment environments and infrastructure setup of the revised Linux command-line learning platform.
It first outlines the deployment goals and the rationale for selecting the University of Tartu HPC Kubernetes environment, Microsoft Azure, and Amazon Web Services as target platforms.
It then presents the setup of each environment, including the provisioning approach, the main deployed components, and the key operational considerations.
Together, these deployment descriptions provide the basis for the cross-platform evaluation of cost, performance, and operational tradeoffs presented in the next chapter.

\subsection{Deployment Goals and Platform Selection} \label{subsec:deployment-goals-and-platform-selection}
The revised Linux command-line learning platform was designed to run across multiple deployment environments while preserving the same core application functionality.
The goal of this multi-platform approach was not portability alone, but also to enable a structured comparison between different hosting models and to identify the most suitable long-term deployment environment for continued use.
For this reason, deployment is discussed separately from the application architecture.
Chapter 3 focused on the internal design and implementation of the system, whereas this chapter examines how the system was deployed in different environments and what operational tradeoffs those environments introduced.

Across all target environments, the deployment design was guided by several shared goals.
First, the platform had to support isolated Linux-based execution environments for individual users, as this is central to the teaching model of the system.
Second, the deployment process needed to become reproducible and maintainable, replacing the more manual setup process of the earlier implementation with a clearer and more structured approach.
Third, the system needed reliable persistence and integration between the application, database, and supporting services.
Finally, because one of the main goals of this thesis is cost optimization, deployment choices also had to account for operational overhead and cost-effectiveness.

Based on these goals, the University of Tartu HPC Kubernetes environment was selected as one of the deployment targets.
This platform was particularly relevant because the original application model was already based on containers and Kubernetes, making the university-managed Kubernetes service a natural continuation of that design.
In addition, UT HPC provides a managed Kubernetes environment inside the University of Tartu network, where users are given access to dedicated namespaces rather than operating the cluster itself.
This made it a meaningful candidate both as a technically compatible platform and as an example of how the system could be hosted within university infrastructure going forward.

Microsoft Azure and Amazon Web Services were selected as the two public cloud platforms for this comparison.
Azure was included because the previous thesis implementation already used Azure, making it the most direct baseline for modernization and cost comparison.
AWS was included to extend the comparison beyond a single provider and to evaluate how well the redesigned system could be deployed in a second major cloud environment.
Together, these three platforms provide a basis for comparing institutional hosting and public cloud deployment models in a consistent manner.

\subsection{University Kubernetes Cluster} \label{subsec:university-kubernetes-cluster}
This subchapter presents the deployment of the revised platform in the University of Tartu HPC Kubernetes environment.
In addition to serving as one of the target platforms in the cross-environment comparison, this deployment is also relevant as a concrete example of how a university-hosted educational application can be deployed in the managed Kubernetes environment provided by UT HPC. The following subchapters first introduce the UT HPC Kubernetes environment and the namespace access process, and then describe how the application was deployed there using Helm and integrated with the supporting services required for operation.

\subsubsection{UT HPC as a Deployment Environment} \label{subsubsec:ut-hpc-as-a-deployment-environment}
The University of Tartu HPC Centre provides multiple research- and teaching-oriented computing services, including a managed Kubernetes service for hosting containerized workloads.
According to the UT HPC service description, the Kubernetes cluster is deployed on bare-metal nodes within the University of Tartu network in a high-availability configuration, while users are granted managed, namespace-level access rather than responsibility for administering the underlying cluster infrastructure~\cite{center_kubernetes_nodate}.
In this model, projects are assigned dedicated namespaces, enabling application deployment and management within a namespace-scoped environment, while broader cluster administration remains outside the scope of ordinary user access~\cite{noauthor_quick_nodate}.

In the context of this thesis, the UT HPC Kubernetes environment was a relevant deployment target for two primary reasons.
Firstly, the revised platform was already based on containerized services and Kubernetes-oriented execution logic, making the managed Kubernetes environment a technically compatible continuation of the application design.
Secondly, deploying the platform within university-managed infrastructure made it possible to evaluate a realistic institutional hosting option for future course use.
As a result, the UT HPC environment was not only one of the platforms included in the cross-environment comparison, but also a concrete university-hosted deployment case for the revised system.

\subsubsection{Access and Namespace Provisioning} \label{subsubsec:access-and-namespace-provisioning}
Access to the UT HPC Kubernetes environment was obtained through the namespace request process described in the UT HPC documentation.
According to the UT HPC Kubernetes quick start guide, access can be requested either by submitting a Kubernetes request or by contacting HPC support directly~\cite{noauthor_quick_nodate}.
As part of the request, the applicant can specify details such as the domain name, deployment lifetime, project repository, and whether the application requires a database~\cite{center_container_nodate}.
After the request has been processed, the user is notified by email and access is granted at the namespace level~\cite{noauthor_quick_nodate}.

As part of this thesis, the application was deployed into the namespace terminal-cs-ut-ee.
After the namespace had been provisioned, cluster access was configured using the kubeconfig file-based access model described in the UT HPC documentation~\cite{noauthor_quick_nodate}.
In addition to the namespace provisioning itself, several deployment-specific integrations were coordinated directly with UT HPC support by email.
These included exposing the application under the public hostname terminal.cs.ut.ee through the UT HPC ingress infrastructure~\cite{noauthor_ingress_nodate}, configuring single sign-on through an HPC-managed Keycloak instance, and configuring SMTP settings required for sending platform validation emails.
UT HPC support also provided a monitoring dashboard for standard workload metrics, including pod CPU usage, memory usage, and the number of active student containers.

\subsubsection{Helm} \label{subsubsec:helm}
The application was deployed to the UT HPC Kubernetes environment using Helm.
Helm is described by its maintainers as a package manager for Kubernetes that simplifies the deployment and lifecycle management of applications through charts~\cite{noauthor_helm_nodate}.
In Helm terminology, a chart is a structured package containing the files needed to describe an application's Kubernetes resources, including templates that are rendered into manifests using supplied configuration values~\cite{noauthor_charts_nodate}.
This made Helm a suitable deployment mechanism for the UT HPC environment, where the cluster infrastructure was already available and the main requirement was to deploy the application's resources within the allocated namespace.

In this project, the Helm chart was used to bundle the Kubernetes resource templates required by the platform and to separate reusable defaults from environment-specific settings.
Helm supports this by allowing the baseline configuration from values.yaml to be overridden through user-provided values files or command-line parameters~\cite{noauthor_values_nodate}.
This made it possible to preserve a common chart structure while adapting settings needed for UT HPC, such as ingress-related options and database configuration.
In practice, deployment to UT HPC mainly involved providing the appropriate environment-specific values and installing the chart in the assigned namespace.
This approach made the setup repeatable and reduced maintenance effort over time.

\subsubsection{Deployed Components and Operational Considerations} \label{subsubsec:deployed-components-and-operational-considerations-helm}
The application was installed in the terminal-cs-ut-ee namespace and made accessible to users via the public hostname terminal.cs.ut.ee.
The Helm release deployed the main web application together with the Kubernetes resources required to operate the platform in this environment.
These included the application Deployment, the public-facing Service and Ingress, the database layer configured using the MariaDB operator, cleanup CronJobs, and additional namespace-level resources such as service accounts, role bindings, and network policies.
The deployment remained fully contained within the designated namespace, while public access was provided through the UT HPC ingress infrastructure~\cite{noauthor_ingress_nodate}.
Figure~\ref{fig:ut-hpc-deployment-overview} presents the UT HPC Kubernetes deployment of the platform, showing the namespace-scoped application components, ingress path, database layer, and per-user containers.

Public access was implemented through a Kubernetes Ingress resource.
UT HPC documentation identifies ingress as a supported method for publishing HTTP applications to the public internet through its managed proxy infrastructure, while routing, DNS records, and TLS certificates were coordinated with HPC support~\cite{noauthor_ingress_nodate}.
Within the namespace, the web application was deployed as a standard Kubernetes Deployment behind a Service, with optional autoscaling available through chart configuration for the UT HPC environment.
Configuration was kept separate from the application code by storing non-sensitive settings in ConfigMaps and providing sensitive values as Secrets.
Service accounts and access control resources were used to define the permissions required by the application and its supporting workloads.
Network policies were additionally used to restrict communication paths between the main application, the database, and the per-user execution environments.

The database layer was deployed using MariaDB, one of the supported database options in the UT HPC Kubernetes environment~\cite{noauthor_databases_nodate}.
This was also appropriate for the project because MariaDB documentation describes MariaDB as having historically functioned as a drop-in replacement for equivalent MySQL versions, making it a compatible continuation of the earlier MySQL-based setup~\cite{noauthor_mariadb_2026}.
In addition to the main web application and database components, the chart deployed periodic cleanup jobs to remove expired session resources and other stale state.
Per-user execution environments were integrated into the same namespace-scoped deployment model, allowing interactive student containers to be created and managed alongside the main platform while remaining within the security and networking boundaries defined for the namespace.

From an operational perspective, the UT HPC deployment reduced the amount of infrastructure that had to be managed directly within the scope of the project.
The Kubernetes cluster and ingress infrastructure were provided by UT HPC, while the responsibility of the application deployment remained focused on namespace-scoped resources such as the Helm release, database setup, and supporting workloads.
At the same time, some parts of the deployment, including public exposure, single sign-on, and SMTP integration, depended on coordination with UT HPC support.
Nevertheless, the environment proved to be a realistic and technically compatible option for future university-hosted use of the platform.

\subsection{Microsoft Azure} \label{subsec:microsoft-azure}
This subchapter discusses the deployment of the revised platform on Microsoft Azure.
Azure was kept as the target environment because the baseline implementation described in chapter~\ref{subsubsec:cloud-deployment-on-azure} was already hosted there, making it the most direct platform for modernization and comparison.
While the University of Tartu HPC deployment shows how the platform can be hosted in institutional Kubernetes infrastructure, the Azure deployment serves as a public-cloud alternative that highlights managed services, infrastructure automation, and lower always-on costs.

\subsubsection{Azure as a Deployment Environment} \label{subsubsec:azure-as-a-deployment-environment}
Microsoft Azure was selected as a deployment target because it provides the clearest continuation of the baseline system and therefore allows the revised design to be compared with the previous Azure-based deployment model.
As described in chapter~\ref{subsubsec:cloud-deployment-on-azure}, the earlier implementation used Azure Container Apps (ACA) for the main web application and Azure Kubernetes Service (AKS) for the Linux execution environments~\cite{eistre_veebipohise_nodate}.
Although this architecture was functional, it still retained some of the cost and operational characteristics of cluster-based orchestration, since the execution environments relied on AKS worker nodes and related infrastructure even when user activity was low~\cite{eistre_veebipohise_nodate}.

For the revised implementation, Azure remained relevant, but the service choices were reconsidered.
The main goal was to preserve the same educational workflow while reducing idle infrastructure overhead and simplifying the operational model.
As a result, the revised Azure deployment moves away from a cluster-centric execution model and instead uses a combination of managed and consumption-oriented services that better fit the platform's intermittent classroom usage pattern.
This made Azure a suitable environment not only for continuity with the baseline system, but also for evaluating whether a more lightweight Azure-native architecture could better support the thesis goals of maintainability and cost optimization.

\subsubsection{Terraform} \label{subsubsec:terraform}
The revised Azure deployment was provisioned using Terraform.
According to the Terraform documentation, Terraform is an infrastructure-as-code (IaC) tool that allows infrastructure to be specified in configuration files and managed through a repeatable workflow~\cite{noauthor_what_nodate-1}.
Compared with the more manual setup process used in the previous system, this allowed the Azure infrastructure to be expressed declaratively.

In this project, Terraform was used to define the Azure infrastructure required by the platform while keeping reusable infrastructure definitions separate from deployment-specific configuration.
Terraform supports this by providing input variables, which make it possible to supply configuration values separately from the core infrastructure code~\cite{noauthor_use_nodate}.
This made it possible to keep a common infrastructure definition while adjusting specific settings for different deployments.
In practice, provisioning the Azure environment mainly involved providing the required variable values and applying the Terraform configuration.
This improved the reproducibility of the deployment process and reduced maintenance effort over time.

\subsubsection{Deployed Components and Operational Considerations} \label{subsubsec:deployed-components-and-operational-considerations-azure}
Similarly to the baseline Azure deployment, the revised Azure deployment is centered around Azure Container Apps (ACA), which hosts the main web application, while the scheduled maintenance workloads are deployed as Azure Container Apps Jobs, which execute containerized tasks for a finite duration and then stop~\cite{noauthor_azure_nodate,craigshoemaker_jobs_nodate}.
In this environment, the main Nuxt application runs as a containerized web workload, while the database cleanup and provisioner cleanup functions run as separate scheduled jobs.
This matches the architectural separation described in Section~\ref{subsec:architecture-overview}, where periodic maintenance was moved out of the main application process.

Figure~\ref{fig:azure-deployment-overview-revised} illustrates the revised Azure deployment.
Users access the platform through the public endpoint provisioned for the application by ACA, which then routes traffic to the main web application container.
When an interactive terminal session is requested, the application provisions a new user-specific Linux container through Azure Container Instances (ACI)~\cite{tomvcassidy_serverless_nodate}.
The use of ACI allows the platform to provide isolated containers for each user without the need to maintain any virtual machines or cluster infrastructure~\cite{tomvcassidy_serverless_nodate}.
Compared with the baseline AKS model, this reduces infrastructure as well as operational overhead as the application does not need to manage cluster runtime for cost optimization.
In this model, the runtime cost of the session containers is determined by the requested vCPU and memory resources together with the duration of the session~\cite{noauthor_pricing_nodate}.

The database layer is implemented using the same Azure Database for MySQL Flexible Server deployment model as in the baseline system, using the cheapest available configuration to minimize costs (B1ms burstable tier with 1 vCPU and 2 GB of memory)~\cite{noauthor_flexible_nodate,eistre_veebipohise_nodate}.
Unlike the baseline implementation, which relied on Azure Files to store per-session SSH keys, the revised deployment uses Azure Key Vault for secret storage~\cite{msmbaldwin_azure_nodate,eistre_veebipohise_nodate}.
Similar to the baseline system, email delivery for account verification is handled through Azure Communication Services using an Azure-managed email domain~\cite{sroons_azure_nodate,bashan-git_overview_nodate,eistre_veebipohise_nodate}.

Regarding authentication, the Azure deployment does not integrate with the university-managed Keycloak service, as that setup is specific to the UT HPC environment.
Accordingly, it follows the same pattern as the baseline Azure system, where users register within the application and validate their accounts via email.
This keeps the Azure deployment self-contained within the cloud environment.
From an operational perspective, the revised Azure deployment differs from the baseline primarily in its reduced dependence on cluster-based infrastructure.
The main web application and the scheduled jobs remain deployable as managed container workloads, while session containers are provisioned on demand through ACI\@.
It also eliminates the need to stop and restart a Kubernetes cluster to control idle costs, avoiding the starting delay when the system is accessed after inactivity.
This design is better aligned with the irregular usage pattern of the learning platform, where demand is concentrated around course sessions rather than spread uniformly over time.
At the same time, Azure still provides a technically compatible and realistic long-term hosting option, especially for cases where an Azure-native deployment path is preferred for administrative or institutional reasons.

\subsection{Amazon Web Services} \label{subsec:amazon-web-services}
This subchapter describes the deployment of the revised platform on Amazon Web Services (AWS).
AWS was included as an additional public cloud platform to complement the Azure-based baseline and to evaluate how the redesigned system can be deployed in a second major cloud environment with a different service ecosystem.
In this way, the AWS deployment broadens the cross-platform comparison by adding another public-cloud reference point for infrastructure automation, operational overhead, and cost drivers.

\subsubsection{AWS as a Deployment Environment} \label{subsubsec:aws-as-a-deployment-environment}
Amazon Web Services was selected as an additional public cloud deployment target to broaden the evaluation beyond a single cloud provider.
Similarly to Azure, AWS provides a realistic environment for assessing how the redesigned platform can be deployed using managed cloud services rather than Kubernetes resources.
Unlike the University of Tartu HPC deployment, which represents institutional Kubernetes hosting, the AWS deployment reflects an alternative public cloud model based on managed services and on-demand resource usage.
This makes AWS a suitable comparison platform for evaluating another major public cloud environment while keeping the core application requirements unchanged.
The AWS infrastructure was provisioned similarly to Azure using Terraform, with the root configuration connecting reusable modules and supplying the required values through input variables.

\subsubsection{Deployed Components and Operational Considerations} \label{subsubsec:deployed-components-and-operational-considerations-aws}
The revised AWS deployment combines an always-on web hosting layer with on-demand and scheduled container workloads.
The main Nuxt web application is deployed to Amazon Lightsail container services, while the database cleanup and provisioner cleanup workloads are deployed as separate ECS tasks that run on Fargate when triggered on a schedule.

Figure~\ref{fig:aws-deployment-overview-revised} illustrates the revised AWS deployment.
Users access the platform through the HTTPS endpoint of the Lightsail container service, which routes requests to the main web application container~\cite{noauthor_lightsail_container_service_deployments_nodate}.
When an interactive terminal session is requested, the application provisions a new user-specific Linux execution environment by launching an Amazon ECS task on AWS Fargate.
Amazon ECS task definitions act as blueprints for running one or more containers, including the container image, runtime configuration, and resource parameters~\cite{noauthor_ecs_task_definitions_nodate}.
AWS documents Fargate as a way to run containers with Amazon ECS without managing servers or EC2-based cluster capacity, while defining CPU, memory, networking, and IAM settings per task~\cite{noauthor_fargate_ecs_nodate}.
In this design, session environments are therefore created on demand as isolated tasks rather than kept running continuously.

Per-session SSH key material is generated by the application, with the private key stored in AWS Secrets Manager and the public key injected into the session container at startup.
AWS describes Secrets Manager as a service for managing and retrieving secrets such as credentials and keys without hard-coding them into application code~\cite{noauthor_secrets_manager_nodate}.

Persistent application data is stored in a managed MySQL database provided by Amazon Lightsail.
In the deployed configuration, the database used the cheapest available Lightsail managed database bundle, priced at \$15 per month, which was covered by the three-month free offer listed on the Lightsail pricing page~\cite{noauthor_lightsail_pricing_nodate}.
Lightsail databases support managed MySQL while reducing administrative effort through built-in operational features such as snapshots and point-in-time recovery~\cite{noauthor_lightsail_databases_nodate}.
To support encrypted database connectivity, the deployment uses the region-specific certificate bundle documented for secure application connectivity to Lightsail databases~\cite{noauthor_lightsail_db_ssl_cert_nodate}.
Regarding authentication, the AWS deployment did not integrate with the university-managed Keycloak service, for the same reason as the Azure deployment.
At the same time, AWS itself did not provide an equivalent managed email-domain setup for this project, so SES was not used for account verification in the deployed configuration.
As a result, the AWS environment represents the weakest authentication model among the evaluated platforms, relying only on application-managed accounts together with the configured email-domain whitelist.

From an operational perspective, the AWS deployment separates predictable always-on cost from usage-dependent execution cost.
For the web layer, the deployment used a Lightsail container service with the Nano plan, corresponding to a \$7 per month node price~\cite{noauthor_lightsail_pricing_nodate}.
For terminal sessions, Fargate pricing depends on the requested vCPU, memory, storage, and runtime duration of the task~\cite{noauthor_fargate_pricing_nodate}.
Scheduled cleanup is implemented through EventBridge scheduled rules that target ECS tasks, which keeps periodic maintenance outside the main application process and aligns it with the same task-based operational model~\cite{noauthor_eventbridge_schedule_nodate,noauthor_eventbridge_targets_nodate}.
Overall, the AWS deployment avoids a continuously running cluster for session execution, but it does so by combining several managed AWS services with different pricing and operational characteristics.
The empirical cost and performance implications of this deployment, and its differences relative to the UT HPC and Azure environments, are analyzed in Chapter~5.
